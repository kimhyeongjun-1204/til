## 🚨 JPA를 모르고 SQL로 DB를 만들면 생기는 일

### 📌 문제 상황

졸업 프로젝트 **AI Groove** 개발 중이었다.
구조는 이랬다.

- **관리자 웹 (내 담당)** : Spring Boot + JPA
- **게임 서버 (팀원 담당)** : 별도 서버
- **DB** : 두 서비스가 **하나의 DB를 공유**

여기서 문제가 터졌다.
나는 `@Entity`로 매핑해서 JPA가 자동으로 테이블을 생성하게 두었고, 게임 서버 팀원은 SQL 스크립트로 직접 테이블을 만들었다.

서로가 **JPA의 동작 방식**과 **명명 규칙**을 충분히 공유하지 않은 채 작업한 결과, DB 충돌이 발생했다.

&nbsp;

---

### 🤔 왜 충돌이 났을까?

#### 원인 1 : `ddl-auto` 설정의 파괴적 동작

당시 내 `application.yml` 설정은 대략 이런 상태였다.

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: create
```

| 옵션 | 동작 |
|------|------|
| `create` | 앱 시작할 때마다 **테이블 전부 DROP 후 재생성** |
| `create-drop` | 위와 같고, 종료 시에도 DROP |
| `update` | 기존 테이블 유지 + 컬럼만 추가 (이름/타입 다르면 충돌) |
| `validate` | 스키마 검증만, 변경 X |
| `none` | 아무것도 안 함 |

즉, 내가 Spring Boot를 재시작할 때마다 **팀원이 SQL로 만든 테이블이 통째로 날아갈 수 있는 상태**였다.

&nbsp;

#### 원인 2 : 명명 규칙 (Naming Strategy) 불일치

JPA는 기본적으로 카멜케이스를 스네이크케이스로 변환한다.

```java
@Entity
public class GameRecord {
    private String userName;          // → user_name
    private LocalDateTime createdAt;  // → created_at
}
```

그런데 팀원이 만든 테이블은 이런 식이었다.

```sql
CREATE TABLE game_record (
    userName VARCHAR(50),    -- 카멜 그대로
    createDate DATETIME       -- 컬럼명 자체도 다름
);
```

JPA는 `user_name` 컬럼을 찾는데, DB에는 `userName`밖에 없으니 **조회할 때마다 컬럼을 찾지 못해 에러 발생**.

&nbsp;

### ⚙️ 진짜 원인은 "소통 부재"

기술적인 원인은 위 두 가지지만, 본질적인 원인은 따로 있었다.

> **"같은 DB를 쓰면서도 서로의 작업 방식을 공유하지 않았다."**

- 나는 팀원이 SQL로 테이블을 직접 만들 거라고 예상하지 못했고,
- 팀원은 내가 JPA로 자동 생성한다는 사실, 그리고 그게 어떤 식으로 동작하는지 몰랐다.

JPA를 모르는 사람 입장에서는 "그냥 자바 객체일 뿐"이라고 생각하기 쉽다.
하지만 JPA는 **DB 스키마까지 직접 만지는 기술**이라는 걸 양쪽 모두 인지했어야 했다.

&nbsp;

---

### ✅ 해결 방법

#### 1. `ddl-auto` 옵션 조정

운영/공유 DB에서는 절대 `create`를 쓰면 안 된다.
실제 협업 환경에서는 다음 두 옵션이 안전하다.

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate   # 스키마 검증만 함, 변경 X
```

| 추천 옵션 | 사용 시점 |
|-----------|-----------|
| `update` | 개인 개발 초기 단계 |
| `validate` | 팀 공유 DB / 운영 환경 |
| `none` | 마이그레이션 도구로 관리할 때 |

&nbsp;

#### 2. 명명 규칙 통일

팀에서 **카멜케이스 vs 스네이크케이스**를 먼저 합의해야 한다.
스네이크케이스로 통일하면 JPA 기본 전략과 자연스럽게 맞는다.

만약 기존 테이블 명명이 다르다면 `@Column`으로 명시적으로 매핑해야 한다.

```java
@Column(name = "userName")
private String userName;
```

&nbsp;

#### 3. DB 스키마를 단일 진실 공급원(SSOT)으로

가장 깔끔한 방법은 **DB 스키마를 SQL 스크립트로 관리하고, JPA는 그걸 따라가게 하는 것**이다.

- `ddl-auto: validate`로 두기
- DDL은 SQL 스크립트 또는 Flyway/Liquibase로 관리
- 모든 팀원이 같은 스크립트를 기준으로 작업

&nbsp;

### 📌 회고

이 경험을 통해 배운 점.

> **"기술을 안다 = 그 기술이 어디까지 영향을 주는지 안다"**

JPA가 단순히 "객체와 DB를 매핑해주는 도구"라고만 알았다면, DB 공유 환경에서 어떤 부작용이 있는지 예상할 수 없었다.
프레임워크는 편하지만, **그 편의가 어디까지 자동으로 처리되는지**를 모르면 협업에서 사고가 난다.

또 하나는 협업 측면이다.
**DB는 공유 자산**이다. 같은 DB를 쓴다면 스키마 변경 방식, 명명 규칙, 마이그레이션 방식을 **반드시 사전에 합의**해야 한다.

> 기술 부족이 아니라 **소통 부족**이 더 큰 장애를 만든다는 걸 절감했다.
