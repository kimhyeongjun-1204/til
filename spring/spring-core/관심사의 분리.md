# 스프링 핵심 원리 - 관심사의 분리와 AppConfig

> 객체의 **생성·연결(구성)** 과 **실행(사용)** 을 분리해서 OCP·DIP를 지키는 과정 정리

&nbsp;

---

## 🚨 문제 상황 — DIP를 지킨 줄 알았는데 아니었다

새로운 정률 할인 정책(`RateDiscountPolicy`)을 적용하려고 `OrderServiceImpl`을 봤더니, 정책을 바꾸려면 **클라이언트 코드를 직접 고쳐야** 했다.

```java
public class OrderServiceImpl implements OrderService {
    // 정책을 바꿀 때마다 이 줄을 수정해야 함
    // private final DiscountPolicy discountPolicy = new FixDiscountPolicy();
    private final DiscountPolicy discountPolicy = new RateDiscountPolicy();
}
```

겉보기엔 `DiscountPolicy` 인터페이스(추상)에 의존하니 DIP를 지킨 것 같지만, 실제로는 `new FixDiscountPolicy()`처럼 **구체 클래스에도 같이 의존**하고 있었다.

| 의존 종류 | 대상 |
| --- | --- |
| 추상(인터페이스) 의존 | `DiscountPolicy` |
| 구체(구현) 클래스 의존 | `FixDiscountPolicy`, `RateDiscountPolicy` |

- **DIP 위반**: 추상뿐 아니라 구체 클래스에도 의존
- **OCP 위반**: 구현체를 바꾸면(`Fix` → `Rate`) 클라이언트 코드까지 변경됨 → "변경에 닫혀 있다"가 깨짐

&nbsp;

## 🤔 시도 — 인터페이스에만 의존하게 바꿨더니 NPE

DIP를 지키려고 구체 클래스 의존을 지우고 인터페이스에만 의존하도록 바꿨다.

```java
public class OrderServiceImpl implements OrderService {
    // 구현체를 직접 정하지 않음
    private DiscountPolicy discountPolicy;
}
```

그런데 구현체가 없으니 실행 시 **`NullPointerException`(NPE)** 발생.

➡️ 즉, **누군가가 대신 구현 객체를 생성해서 주입해줘야** 한다는 결론에 도달.

&nbsp;

## ✅ 해결 — 관심사의 분리 (공연 기획자 비유)

애플리케이션을 하나의 **공연**이라고 생각해보자.

- 인터페이스 = **배역**(배우 역할)
- 구현체 = 그 배역을 맡은 **배우**

로미오와 줄리엣 공연에서, 로미오 역의 배우가 줄리엣 역 배우까지 직접 섭외하면 안 된다. 그런데 기존 코드는 딱 그 꼴이었다 — `OrderServiceImpl`(배우)이 자기 로직도 실행하고, 어떤 할인 정책을 쓸지(배우 섭외)까지 다 책임짐.

➡️ **배우는 연기만**, 섭외·배역 지정은 **공연 기획자**가 맡아야 한다.
➡️ 이 공연 기획자 역할을 하는 게 바로 **`AppConfig`** (아직 스프링 아님! 순수 자바 단계).

&nbsp;

### 📌 AppConfig 등장

애플리케이션 전체 동작 방식을 **구성(config)** 하기 위해, 구현 객체를 **생성하고 연결**하는 책임만 가지는 설정 클래스.

```java
public class AppConfig {

    public MemberService memberService() {
        return new MemberServiceImpl(new MemoryMemberRepository());
    }

    public OrderService orderService() {
        return new OrderServiceImpl(
                new MemoryMemberRepository(),
                new FixDiscountPolicy());
    }
}
```

`OrderServiceImpl`은 생성자로 의존성을 받기만 한다. 어떤 구현체가 들어올지는 **오직 외부(AppConfig)가 결정**한다.

```java
public class OrderServiceImpl implements OrderService {
    private final MemberRepository memberRepository;
    private final DiscountPolicy discountPolicy;

    // 어떤 구현체가 주입될지 본인은 모름 → 실행에만 집중
    public OrderServiceImpl(MemberRepository memberRepository, DiscountPolicy discountPolicy) {
        this.memberRepository = memberRepository;
        this.discountPolicy = discountPolicy;
    }
}
```

## ⚙️ AppConfig 리팩터링 — 중복 제거 + 역할 드러내기

리팩터링 전에는 `new MemoryMemberRepository()`가 두 번 생성되는 **중복**이 있고, 역할 구분도 안 보였다. 메서드로 빼주면 깔끔해진다.

```java
public class AppConfig {

    public MemberService memberService() {
        return new MemberServiceImpl(memberRepository());
    }

    public OrderService orderService() {
        return new OrderServiceImpl(memberRepository(), discountPolicy());
    }

    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }

    public DiscountPolicy discountPolicy() {
        return new FixDiscountPolicy();
    }
}
```

- `new MemoryMemberRepository()` 중복 제거 → 구현체 변경 시 **한 곳만** 수정
- `memberService`는 `memberRepository`를 쓰고, `orderService`는 `memberRepository` + `discountPolicy`를 쓴다는 **역할과 구현이 한눈에** 보임

## 📌 사용 영역 vs 구성 영역

AppConfig 덕분에 애플리케이션이 두 영역으로 분리됐다.

| 구분 | 담당 | 예시 |
| --- | --- | --- |
| **구성 영역** | 어떤 구현체를 쓸지, 누구에게 뭘 주입할지 결정 | `AppConfig` |
| **사용 영역** | 주입받은 의존성으로 비즈니스 로직만 실행 | `MemberServiceImpl`, `OrderServiceImpl` |

그래서 할인 정책을 `Fix` → `Rate`로 바꿔도 **구성 영역(AppConfig) 한 줄만** 고치면 끝. 사용 영역은 한 글자도 안 건드린다.

```java
public DiscountPolicy discountPolicy() {
    // return new FixDiscountPolicy();
    return new RateDiscountPolicy(); // 여기만 변경
}
```

## ✅ 적용된 객체지향 5원칙 (SRP·DIP·OCP)

| 원칙 | 적용 내용 |
| --- | --- |
| **SRP** (단일 책임) | 생성·연결은 AppConfig가, 실행은 클라이언트가 담당하도록 책임 분리 |
| **DIP** (의존관계 역전) | 클라이언트는 추상(`DiscountPolicy`)에만 의존, 구현체 생성·주입은 AppConfig가 대신 |
| **OCP** (개방-폐쇄) | 구현체를 확장·변경해도 사용 영역 코드는 변경되지 않음(변경에 닫힘) |

&nbsp;

## 📌 IoC / DI / 컨테이너

- **IoC (제어의 역전)**: 구현 객체가 스스로 흐름을 제어하던 것을, 외부(AppConfig)가 가져감. `OrderServiceImpl`은 어떤 구현체가 실행될지도 모른 채 자기 로직만 실행.
- **DI (의존관계 주입)**: 런타임에 외부에서 실제 구현 객체를 생성해 주입. 정적 클래스 의존관계는 그대로 두고, 동적 객체 의존관계만 쉽게 변경 가능.
- **IoC/DI 컨테이너**: AppConfig처럼 객체를 생성·관리하며 의존관계를 연결해주는 것. (다음 단계에서 스프링 `ApplicationContext`가 이 역할을 가져감)

&nbsp;

---

> **핵심 요약**
> AppConfig는 "공연 기획자"로서 **객체의 생성과 연결(구성)** 을 전담하고, 각 클라이언트(배우)는 **실행** 에만 집중한다. 이 구성/사용 분리가 곧 SRP·DIP·OCP를 지키는 방법이며, 다음 단계에서 이 AppConfig에 `@Configuration` + `@Bean`을 붙여 스프링 컨테이너로 넘긴다.
