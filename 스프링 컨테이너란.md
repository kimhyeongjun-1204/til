## 🌱 스프링 컨테이너 


### 📌 스프링 컨테이너의 역할 
: 빈의 생성, 의존성 주입, 생명주기를 관리하는 스프링의 핵심 객체. 
: 코드 상으로는 ApplicationContext 인터페이스가 곧 스프링 컨테이너. 

&nbsp;

### 📌 빈 생성 방법 
: @Component와 @Bean 어노테이션을 통해 빈 등록

&nbsp;

### ✅ @Component와 @Bean의 차이 
* @Component의 특징 : 1. 클래스 단위에서 사용됨  2. 개발자가 직접 작성한 클래스에 사용 
* @Bean의 특징 : 1. 메소드 단위에서 사용 2. 소스코드를 수정할 수 없는 외부 라이브러리 객체를 등록할 때 사용

&nbsp;

### 📌 빈 생성 시점 
: 빈은 애플리케이션 시작 시점에 생성되고 주입까지 완료됨 

&nbsp;

### 🤔 스프링이 빈을 싱글톤으로 관리하는 핵심 이유 
=> 오버헤드 방지가 아닌 의존성 관리(DI)와 객체 간 협력 구조를 프레임워크가 제어하기 위해서

&nbsp;

## ⚙️ 동작 과정 

&nbsp;

### 1️⃣ 빈 정의(BeanDefinition) 수집  
: @ComponentScan으로 클래스패스를 스캔, @Configuration 클래스의 @Bean 메서드를 파싱하여 어떤 빈을 어떻게 만들어야 하는지에 대한 메타정보를 수집

&nbsp;

### 2️⃣ 빈 인스턴스 생성 
: 수집된 BeanDefinition을 기반으로 실제 객체를 생성. 기본 스코프가 싱글톤이므로 컨테이너당 하나의 인스턴스만 생성하여 싱글톤 레지스트리에 보관

&nbsp;

### 3️⃣ 의존성 주입(DI) 
: 생성된 빈들 사이의 의존 관계를 해결 

```java
@Service 
public class OrderService {

    private final OrderRepository orderRepository;

    // 생성자 주입 : 권장 
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository; 
    }
}
```

위의 코드에서 원래는 개발자가 new OrderRepository() 를 호출하여 객체를 생성해야 하지만 Spring에서는 컨테이너가 이미 만들어둔 OrderRepository 빈을 찾아서 주입해줌 

&nbsp;

> 🚨 생성자 주입 이외에도 필드 주입과 세터 주입이 있지만 생성자 주입을 권장 

&nbsp;

### 4️⃣ 초기화 콜백 
: @PostConstruct 등 초기화 로직이 있으면 이 시점에 실행 

&nbsp;

## 📌 빈에 등록한다란 

스프링 컨테이너(ApplicationContext)가 해당 객체의 생성 · 의존성 주입 · 생명주기 관리를 대신 관리하도록 맡기는 것. 

원래 개발자가 직접 new로 만들고 관리를 해야 하지만 빈으로 등록하여 책임을 컨테이너에 위임 = **제어의 역전(IoC)**

&nbsp;

---

&nbsp;

## 🤔 스프링 컨테이너에게 책임을 위임하는 이유 

객체 간의 관계에서 당연히 의존성이 생길 수밖에 없는데 개발자가 필요한 객체를 직접 생성하면 **강한 결합**이 생겨남 

&nbsp;

```java
// 🚨 직접 생성 — 강한 결합
@Service
public class OrderService {
    private final OrderRepository orderRepository = new JpaOrderRepository();
}
```

위의 OrderService가 JpaOrderRepository라는 구체 클래스에 묶임. 
이후 MyBatisOrderRepository로 변경하고 싶으면 OrderService의 코드를 변경해야 됨 

&nbsp;

```java
// ✅ 의존성 주입 — 느슨한 결합
@Service
public class OrderService {
    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

OrderService는 OrderRepository **인터페이스**에만 의존. 
JpaOrderRepository든 MyBatisOrderRepository든 스프링 컨테이너가 알아서 넣어주므로 OrderService 코드 수정 없이 구현체 교체 가능
